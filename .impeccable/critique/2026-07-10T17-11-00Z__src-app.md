A trustworthy, well-built core dragged down by one component (the thumbnail) and two omissions (sign-out, a rendered error path outside the composer).

## Anti-Patterns Verdict

**Not slop.** A user fluent in Linear/Raycast/Notion would trust the core. The Three Voices split is genuinely implemented (serif titles `stash.tsx:492/497`, mono commands+URL, sans chrome), the cool-neutral palette is honest to DESIGN.md with no warm tint, the accent is truly rationed, and delete-with-undo obeys the "don't reach for a modal" rule.

**Deterministic scan:** clean — no side-stripe borders, no gradient text, no glassmorphism, no hardcoded z-index, the one `<img>` is correctly `alt=""`+`aria-hidden`, every button has an accessible name, every `onClick` is on a real `<button>`.

The one place it drifts toward category slop — hard — is the just-added website-preview thumbnail (`Thumb`, `stash.tsx:543-567`). It reintroduces the Pocket/read-it-later screenshot aesthetic PRODUCT.md bans by name, and it does real damage (below).

## What's Working

1. **The empty state teaches (`stash.tsx:626-648`).** Defines both object types in one sentence each, shows both shortcuts as `kbd` chips, one low-pressure CTA. Exactly DESIGN.md's brief, and almost nobody ships it.
2. **Delete is reversible, not confirmed (`stash.tsx:194-201`, `310-324`).** Optimistic removal + `role="status"` undo toast honors "capture costs nothing" and gives SR users an announced recovery.
3. **The system ships its rules, not just its manifesto.** Contrast tokens are honest — ink 16.6:1, muted 5.7:1, accent 6.9:1, placeholder held to muted (5.7:1) not a light gray. Flat-at-rest respected: the only shadow is the floating toast.

## Priority Issues

**[P0] Every saved URL is sent to a third party.**
`shotOf` (`stash.tsx:51-53`) builds `s.wordpress.com/mshots/v1/${url}` from the full normalized URL — query string and any embedded token included — and `Thumb` requests it on every render of every tool row. PRODUCT.md's whole thesis is trust for "a small circle of trusted people — not strangers." Silently shipping private/internal URLs to Automattic is undisclosed and directly contradicts the "still be here in three years" promise. Outranks every visual issue.
*Fix:* remove `Thumb`, or gate it behind an explicit per-item opt-in, or self-host. At minimum never transmit the query string. Given the register, deletion is the honest default. → `/impeccable distill`

**[P1] No sign-out exists anywhere.**
The header renders the user's name as static muted text (`stash.tsx:216-219`) with no adjacent control. Auth exists "so a stash follows you across devices," but a user on a shared device can't end the session. Nielsen #3 + a real security gap.
*Fix:* sign-out button beside the name calling `authClient.signOut()` → `/login`; sans label, ghost styling. → `/impeccable harden`

**[P1] Delete/undo errors are silently swallowed.**
`error` is only rendered inside `Composer`'s open-form branch (`stash.tsx:437-442`); the `!open` branch (`352-359`) never reads it. But `onDelete`/`onUndo` (`194-212`) run with the composer closed and call `setError(result.error)` — so a failed delete or failed undo writes to a state var with no DOM output. The user sees nothing. Confirmed by reading both branches.
*Fix:* render the error (and/or the undo failure) at the list level, not only inside the composer. → `/impeccable harden`

**[P1] The thumbnail has no loading state and breaks the title column.**
`Thumb` renders a raw `<img>` with no skeleton. mshots returns a **200 grey image** (not an error) while rendering, so `onError` never fires and an unshootable/private URL shows a permanent grey box. Separately, `Thumb` returns `null` for formulae, so tool rows indent their titles ~120px+gap while formula rows start at the left edge — the aligned "card catalog" column is broken, drifting toward the banned card look. `hidden sm:block` means mobile pays the full privacy cost for zero pixels.
*Fix (if it survives P0):* reserve identical leading space for all rows, render a labelled skeleton while pending, cap the pending state so a never-arriving shot collapses to the no-thumbnail layout. → `/impeccable harden`

**[P2] Three different "selected" treatments for the same control.**
Kind tabs: `bg-surface font-semibold text-ink`, no check (`239-243`). Composer's tool/formula toggle — the *same* choice — `bg-accent text-bg` + ✓ (`382-388`). Tag chips: a third variant (`261-268`). The product register bans exactly this. No element in the app defines an `:active` state either (grep: zero `active:`).
*Fix:* one segmented-control selected treatment applied to both. → `/impeccable polish`

## Persona Red Flags

**Alex (power user):** saves three things, can't find them — list is *always* alphabetical-by-title (`page.tsx:20`, `byTitle` `stash.tsx:25`), no recency. No `⌘K` palette despite the "card catalog" north star. Composer Save has no pending/disabled state (`453-458`), so a double-click can fire `createItem` twice.

**Sam (SR + keyboard):** focus lost on cancel and on delete — the `<li>` unmounts and focus drops to `<body>` (`closeComposer` `112-116` never restores the trigger). Delete is announced but focus never reaches Undo, so recovering a mistaken delete means blind-navigating to a toast that never auto-dismisses. Sub-24px controls (below) fail WCAG 2.2 AA 2.5.8.

**Riley (stress tester):** saves a localhost/internal URL → mshots returns a grey 200, `onError` never fires, permanent grey box. Deletes rapidly near the 60/hr ceiling then hits Undo → `onUndo`→`createItem` is subject to the same limit (`actions.ts:100-114`) and can fail — the undone item is now permanently gone, *and* that failure is swallowed (P1 above). Saves `javascript:` → server rejects, but the optimistic row flashed the raw string first (`draftFrom:62`).

**The Maintainer (project persona):** saves a client dashboard URL with a session token in the query string; transmitted verbatim to `s.wordpress.com` (P0). This is the persona the product exists for, aimed straight at their one stated fear.

## Minor Observations

- **Touch targets:** measured against WCAG 2.2 AA 2.5.8 (24×24, not the 44 AAA bar) — three controls miss: Undo (`319`), "Clear the filters" (`618`), the auth mode-switch link (`auth-form.tsx:105`), all ~16–20px tall with no padding; Copy (`593`) and tag chips (`261`) are borderline. Everything else clears 24.
- **Border contrast:** `border-border #d3dad9` on bg is **1.32:1** — fails WCAG 1.4.11 non-text (needs 3:1) for input/row boundaries that carry state.
- No route-level `error.tsx` in `src/app` — no error boundary; `loading.tsx` is the only special file besides `page.tsx`.
- No recency retrieval axis at all — the only order is alphabetical-by-title. For a product measured by re-visits, that helps only a user who already remembers the exact title.
- The tag filter row is unbounded (`251-274`) — no cap, no overflow. Cognitive-load: fails chunking + minimal-choices (2 failures → moderate) for a heavy tagger.
- "Formulae" (`245`) is precious for a "library card" voice; "Formulas" reads plainer.
- Undo re-creates with a fresh id/timestamp (`203-212`) — invisible only because no dates are shown.

## Questions to Consider

1. If the rule is *the content supplies the color*, is a third-party screenshot the user's content — or the interface faking content and calling that content-first?
2. Success is re-visits, not saves — so why is the sole retrieval order the one axis (alphabetical-by-title) that only helps someone who already remembers the words?
3. A card catalog is one clean column of aligned titles; tool rows indent 120px and formula rows don't. Is this a list, or a card grid that hasn't admitted it yet?
